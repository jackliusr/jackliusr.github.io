== Introduction

For several years, I used Conventional Commits in my software projects.

The idea was simple and practical: write commit messages in a structured format, then use tools to generate changelogs and release notes automatically.

For example:

[source,text]
----
feat: add user login
fix: resolve payment retry issue
docs: update API usage guide
----

This worked well. Tools such as Conventional Changelog could read these commit messages and generate release notes based on the commit history.

For a long time, I thought this was the status quo.

== The Conventional Commits Era

Conventional Commits brought discipline to commit messages.

Instead of writing vague messages like:

[source,text]
----
update code
fix bug
change stuff
----

developers could write more meaningful and machine-readable messages:

[source,text]
----
feat(payment): support offline transaction queue
fix(auth): handle expired refresh token correctly
refactor(api): simplify error handling
----

This made automation easier.

Release notes could be grouped by type:

* Features
* Bug fixes
* Refactoring
* Documentation updates
* Breaking changes

For engineering teams, this was a big improvement. It reduced manual effort and made the release process more consistent.

== The Limitation

However, Conventional Commits still depend heavily on the quality of commit messages.

If the commit message is too short, the generated changelog will also be shallow.

For example:

[source,text]
----
fix: update validation
----

This does not tell users much.

What validation was fixed?

Why did it matter?

Who is affected?

Is there any migration required?

Traditional changelog generation tools can format commit messages nicely, but they usually cannot deeply understand the real context behind the change.

They are good at structure, but limited in explanation.

== The Moment I Changed My View

Recently, I was searching for:

[source,text]
----
how to make llm generate thorough notes
----

During that search, I found SmartNote. It appeared as the top result on the search result page.

That made me think again about release notes.

If an LLM can generate thorough notes from raw information, then why should release notes only be generated from structured commit messages?

Maybe release notes can be much better.

== LLM Changed Everything

LLMs changed the way we think about documentation.

Instead of only converting commit messages into a changelog, an LLM can help explain:

* what changed
* why it changed
* how it affects users
* what developers need to know
* whether there are breaking changes
* what migration steps are required
* what risks or limitations exist

This is very different from traditional changelog generation.

A conventional changelog may say:

[source,text]
----
feat: add batch payment support
----

A better LLM-generated release note could say:

[source,text]
----
This release adds batch payment support, allowing users to submit multiple payment instructions in a single request. This reduces manual operation and improves processing efficiency for high-volume payment scenarios.

Developers should review the new batch payment API contract and ensure proper error handling for partial failures.
----

The second version is more useful to product users, developers, support teams, and stakeholders.

== Conventional Commits Are Still Useful

This does not mean Conventional Commits are useless.

Actually, they are still valuable.

Structured commits provide a strong input signal for automation. They help the LLM understand the type and scope of changes.

A good future workflow may look like this:

[source,text]
----
Conventional Commits
        +
Pull Requests
        +
Issue descriptions
        +
Code diff
        +
Documentation updates
        ↓
LLM-generated release notes
----

In this model, Conventional Commits are not the final output.

They become one important input for a smarter release-note generation process.

== From Changelog to Communication

A changelog is usually written for developers.

Release notes should be written for humans.

That includes:

* developers
* product managers
* support engineers
* business stakeholders
* end users

Different audiences need different levels of detail.

A developer may care about APIs, configuration, and migration.

A business stakeholder may care about value, risk, and impact.

A user may only care about what is new and how it helps them.

LLMs are useful because they can transform the same technical changes into different versions of release notes for different audiences.

== A Better Release Notes Workflow

A modern LLM-based release notes workflow could include the following steps:

. Collect commits, pull requests, issues, and code changes.
. Ask the LLM to group changes by theme.
. Generate a user-friendly summary.
. Generate a developer-focused technical section.
. Detect possible breaking changes.
. Add migration notes where needed.
. Let humans review and edit the final result.

This combines automation with human judgement.

The LLM does the heavy drafting work, but the team still owns the final message.

== Example Prompt

A simple prompt could look like this:

[source,text]
----
Generate release notes based on the following commits, pull requests, and issue descriptions.

Please include:
- Executive summary
- New features
- Bug fixes
- Breaking changes
- Migration notes
- Known limitations
- Developer impact
- User impact

Write clearly for both technical and non-technical readers.
----

This kind of prompt can produce much richer output than a traditional changelog tool.

== My Current Thinking

Several years ago, I believed Conventional Commits plus Conventional Changelog represented the best practice.

Now I think they are only part of the solution.

LLMs make it possible to generate release notes that are more complete, more readable, and more useful.

The status quo is changing.

Today, tools like SmartNote show one possible direction. Tomorrow, better options may appear.

The important point is not one specific tool.

The important point is the shift:

[source,text]
----
From structured commit messages
to intelligent release communication.
----

== Conclusion

Conventional Commits helped teams standardise commit messages and automate changelog generation.

That was a meaningful step forward.

But LLMs have changed the expectation.

Release notes no longer need to be thin summaries of commit messages. They can become thoughtful explanations of product change, technical impact, and user value.

Maybe Conventional Commits were the old foundation.

LLM-generated release notes may become the next layer.

And as better tools appear in the future, the way we write changelogs and release notes will continue to evolve.
